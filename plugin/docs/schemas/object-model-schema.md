# Custom Object Schema Contract

**Document type:** Authoritative contract for plugin-managed content objects (spec §10).  
**Governs:** Object keys, statuses, relationships, lifecycle rules before any CPT registration or repository implementation.  
**Related:** storage-strategy-matrix.md (storage primitive per object); profile-schema.md, profile-snapshot-schema.md.

---

## 1. Internal key and identifier strategy

- **CPT-backed objects:** Each object has a stable **internal key** (slug or machine name) used in code and references, plus a WordPress post ID where the object is stored as a CPT. The internal key is immutable for the lifetime of the object; it is used in relationships and export. For section/page templates and prompt packs, the key is the primary human-facing identifier (e.g. `hero_section_v1`, `landing_page_contact`).
- **Compositions and Build Plans:** Unique **composition id** / **plan id** (e.g. UUID or prefixed id) plus optional human-readable name. ID is immutable.
- **AI Run:** **Run id** (e.g. UUID or prefixed id). Immutable.
- **Documentation / Version Snapshot:** **Document id** / **snapshot id** plus scope reference. Immutable.
- **No arbitrary IDs:** All identifiers are generated by the system or assigned at creation; no user-supplied freeform IDs for object identity.

## 2. Status vocabulary

### 2.1 Shared statuses (across multiple object types)

| Status | Meaning | Use on |
|--------|---------|--------|
| `draft` | Editable, not yet published or activated | Section template, Page template, Composition, Prompt pack, Documentation |
| `active` | In use; visible to planning/execution | Section template, Page template, Composition, Prompt pack, Documentation |
| `inactive` | Not in use; can be reactivated | Section template, Page template, Prompt pack |
| `archived` | Retained but not selectable for new use | Documentation, Version snapshot |
| `deprecated` | Discouraged for new use; may still be referenced | Section template, Page template, Prompt pack |

### 2.2 Object-specific statuses

| Object | Status | Meaning |
|--------|--------|---------|
| **Composition** | `validation_failed` | Fails validation rules |
| **Composition** | `valid` | Passes validation |
| **Build Plan** | `pending_review` | Awaiting user approval/denial |
| **Build Plan** | `approved` | User has approved (or partial approval) |
| **Build Plan** | `rejected` | User has rejected |
| **Build Plan** | `in_progress` | Execution in progress |
| **Build Plan** | `completed` | Execution completed |
| **Build Plan** | `superseded` | Replaced by a newer plan |
| **AI Run** | `pending_generation` | Request queued or in progress |
| **AI Run** | `completed` | Run finished successfully |
| **AI Run** | `failed_validation` | Output failed validation |
| **AI Run** | `failed` | Provider or system failure |
| **Version Snapshot** | `active` | Snapshot is current reference |
| **Version Snapshot** | `superseded` | Newer snapshot exists |

No status values such as `misc`, `done`, or other arbitrary labels. Only the enums above (and any future explicitly added values) are valid.

## 3. Per-object schema tables

### 3.1 Section Template (§10.1)

| Attribute | Value |
|-----------|--------|
| **Object name** | Section Template |
| **Proposed storage** | CPT |
| **Internal key** | Stable slug (e.g. `hero_section_v1`); immutable |
| **Required fields** | internal_key, name, purpose_description, category_or_classification, version_metadata, status |
| **Optional fields** | structural_blueprint_ref, field_group_blueprint_ref, css_contract_ref, helper_paragraph_ref, compatibility_metadata |
| **Status enum** | `draft` \| `active` \| `inactive` \| `deprecated` |
| **Relationships** | Referenced by Page Template (ordered list); by Composition (ordered section list); by Documentation (source). Belongs to template registry (logical grouping). |
| **Lifecycle** | Created as draft; may transition to active → inactive/deprecated. Deprecated may not be used for new compositions but may remain referenced. |
| **Deletion/archive** | Soft-deprecate preferred; hard delete only when no references remain or with explicit cascade policy. |
| **Export** | Exportable; include key, name, version, status, and metadata. |
| **Migration** | Version metadata and compatibility metadata support schema upgrades. |
| **Sensitivity** | Admin-visible; no secrets. |

For the full implementation-grade section schema (required/optional fields, sub-manifests, validation, completeness rules), see **section-registry-schema.md**.

### 3.2 Page Template (§10.2)

| Attribute | Value |
|-----------|--------|
| **Object name** | Page Template |
| **Proposed storage** | CPT |
| **Internal key** | Stable slug (e.g. `landing_contact`); immutable |
| **Required fields** | internal_key, name, intended_use_or_archetype, ordered_section_template_refs, version_metadata, status |
| **Optional fields** | optional_vs_required_section_metadata, compatibility_and_purpose_tags, one_pager_generation_metadata, seo_default_ref |
| **Status enum** | `draft` \| `active` \| `inactive` \| `deprecated` |
| **Relationships** | References Section Templates (ordered list). May be referenced by Composition (source). May be referenced by Documentation. |
| **Lifecycle** | Draft → active → inactive/deprecated. Section refs must resolve to valid section template keys. |
| **Deletion/archive** | Deprecate before delete; handle references from compositions. |
| **Export** | Exportable. |
| **Migration** | Version metadata; section refs are keys, stable across exports. |
| **Sensitivity** | Admin-visible. |

For the full implementation-grade page template schema (required/optional fields, ordered sections, one-pager metadata, compatibility, version/deprecation), see **page-template-registry-schema.md**.

### 3.3 Custom Template Composition (§10.3)

| Attribute | Value |
|-----------|--------|
| **Object name** | Custom Template Composition |
| **Proposed storage** | CPT |
| **Internal key** | Unique composition id (e.g. UUID); immutable |
| **Required fields** | composition_id, name, ordered_section_list, status, validation_status |
| **Optional fields** | source_template_ref (if derived from page template), helper_one_pager_ref, registry_snapshot_ref_at_creation, version_markers |
| **Status enum** | `draft` \| `active` \| `archived` |
| **Validation status** | `valid` \| `validation_failed` (object-specific; stored separately from lifecycle status) |
| **Relationships** | References Section Templates (ordered). May reference Page Template (source). May reference Documentation (one-pager). May reference Version Snapshot (registry at creation). |
| **Lifecycle** | Created as draft; validation_status updated independently. draft/active → archived. |
| **Deletion/archive** | May be archived; hard delete per policy. |
| **Export** | Exportable. |
| **Migration** | Snapshot reference preserves registry state at creation. |
| **Sensitivity** | Admin-visible. |

For the composition validation state machine (lifecycle statuses, validation result, validation codes, transitions, provenance, snapshot and duplication/import rules), see **composition-validation-state-machine.md**.

### 3.4 Build Plan (§10.4)

| Attribute | Value |
|-----------|--------|
| **Object name** | Build Plan |
| **Proposed storage** | CPT |
| **Internal key** | plan_id (e.g. UUID); immutable |
| **Required fields** | plan_id, ai_run_ref, normalized_output_ref, plan_title, plan_summary, site_purpose_summary, site_flow_summary, steps, created_at (see **build-plan-schema.md** for full root, step, and item schema). |
| **Optional fields** | profile_context_ref, crawl_snapshot_ref, registry_snapshot_ref, affected_page_refs, completed_at, actor_refs, approval_denial_state, remaining_work_status, execution_status, warnings, assumptions, confidence, execution_history_anchor, history_retention, schema_version |
| **Status enum** | `pending_review` \| `approved` \| `rejected` \| `in_progress` \| `completed` \| `superseded` |
| **Relationships** | References AI Run (one). References normalized output (one). May reference profile snapshot, crawl snapshot, registry snapshot. May reference pages (created or affected). |
| **Lifecycle** | Created in pending_review. Allowed transitions: pending_review → approved \| rejected; approved → in_progress → completed \| superseded. rejected and superseded are terminal for workflow. |
| **Deletion/archive** | Operational record; archive or retain per policy. Do not delete while execution or rollback depends on it. |
| **Export** | Exportable (operational state exportable under permissions). |
| **Migration** | Timestamps and status markers; ai_run_ref and normalized_output_ref are stable. |
| **Sensitivity** | **Tighter access:** Approval state and execution status are operational; restrict to users with appropriate capabilities. No raw provider secrets in plan. |

**Full schema:** Implementation-grade root, step, and item schema (source references, step grouping, item dependency/blocking, warning/confidence, completion/history) are defined in **build-plan-schema.md**. Code-level constants: `AIOPageBuilder\Domain\BuildPlan\Schema\Build_Plan_Schema`, `Build_Plan_Item_Schema`.

### 3.5 AI Run (§10.5)

| Attribute | Value |
|-----------|--------|
| **Object name** | AI Run |
| **Proposed storage** | CPT (metadata/identity); raw response and artifacts in custom table |
| **Internal key** | run_id (e.g. UUID); immutable |
| **Required fields** | run_id, provider_id, model_id, prompt_pack_version_ref, validation_status, created_at |
| **Optional fields** | input_snapshot_refs, registry_snapshot_refs, crawl_snapshot_refs, raw_response_ref (table id), normalized_output_ref (table id), usage_metadata, timing_metadata, actor_refs |
| **Status enum** | `pending_generation` \| `completed` \| `failed_validation` \| `failed` |
| **Relationships** | References Prompt Pack (version). References profile/snapshot, registry snapshot, crawl snapshot. References artifact records (custom table). Referenced by Build Plan (one-to-one for plan-created-from-run). |
| **Lifecycle** | Created as pending_generation. → completed \| failed_validation \| failed. No status rollback. |
| **Deletion/archive** | Long-lived operational; retain for audit. Archive/delete per retention policy only. |
| **Export** | Exportable (optional; artifact refs may point to redacted or excluded content). |
| **Migration** | run_id stable; refs to tables versioned. |
| **Sensitivity** | **Tighter access:** AI run metadata and artifact references are privileged. Raw provider responses and payloads must not contain secrets; access via capability (e.g. aio_view_sensitive_diagnostics). |

### 3.6 Prompt Pack (§10.6)

| Attribute | Value |
|-----------|--------|
| **Object name** | Prompt Pack |
| **Proposed storage** | CPT |
| **Internal key** | prompt_pack_id (slug or versioned key, e.g. `planning_v1`); immutable per version |
| **Required fields** | prompt_pack_id, name, version, status, system_prompt_components |
| **Optional fields** | contextual_injection_rules, schema_expectations, provider_notes_or_compatibility, changelog_refs |
| **Status enum** | `draft` \| `active` \| `inactive` \| `deprecated` |
| **Relationships** | Referenced by AI Run (version). May reference Version Snapshot (snapshot of pack at version). |
| **Lifecycle** | draft → active. active → inactive/deprecated. New version = new object or version field; no in-place overwrite of active. |
| **Deletion/archive** | Deprecate before delete; ensure no active AI runs depend on deleted version. |
| **Export** | Exportable. |
| **Migration** | Version and changelog support traceability. |
| **Sensitivity** | Admin-visible; prompt content may be sensitive to business logic but not secrets. |

### 3.8 Version Snapshot (§10.8)

| Attribute | Value |
|-----------|--------|
| **Object name** | Version Snapshot |
| **Proposed storage** | CPT or custom table (by scope); implementation may key by scope_type + scope_id |
| **Internal key** | snapshot_id (e.g. UUID); immutable |
| **Required fields** | snapshot_id, scope_type, scope_id, created_at, schema_version |
| **Optional fields** | payload_ref (e.g. serialized ref or table id), object_refs (what this snapshot preserves) |
| **Status enum** | `active` \| `superseded` |
| **Relationships** | References the object(s) it preserves (template registry, prompt pack, profile, build context, etc.). Referenced by Composition (registry at creation), AI Run (input/registry snapshots). |
| **Lifecycle** | Created active. When a new snapshot for same scope is created, previous may be marked superseded. |
| **Deletion/archive** | Retain for traceability; delete per retention policy. |
| **Export** | Exportable as part of context (optional). |
| **Migration** | schema_version and scope identify snapshot; payload format versioned. |
| **Sensitivity** | Depends on payload; profile/registry snapshots admin-visible. |

For the full version snapshot object schema (scope types, root fields, object_refs, version/provenance/compatibility blocks, retention, examples), see **version-snapshot-schema.md**.

## 4. Relationship map

```
Section Template ──┬── referenced by ──► Page Template (ordered list)
                   ├── referenced by ──► Composition (ordered list)
                   └── referenced by ──► Documentation (source)

Page Template ─────┬── references ──► Section Templates
                   ├── referenced by ──► Composition (source)
                   └── referenced by ──► Documentation (source)

Composition ────────┬── references ──► Section Templates, optional Page Template
                   ├── referenced by ──► Documentation (source)
                   └── may reference ──► Version Snapshot (registry at creation)

Build Plan ────────► references ──► AI Run (one)
                   ► may reference ──► Pages, Profile snapshot

AI Run ────────────► references ──► Prompt Pack (version), input/registry/crawl snapshots, artifact records (table)
                   ◄── referenced by ──► Build Plan

Prompt Pack ───────► referenced by ──► AI Run
                   ► may reference ──► Version Snapshot

Documentation ─────► may reference ──► Section Template | Page Template | Composition

Version Snapshot ──► references ──► object(s) it preserves (registry, pack, profile, etc.)
                   ◄── referenced by ──► Composition, AI Run
```

## 5. Lifecycle transition rules (summary)

| Object | Valid transitions | Invalid (examples) |
|--------|-------------------|---------------------|
| Section Template | draft→active, active→inactive, active→deprecated, inactive→active | deprecated→active (without explicit undeprecate policy) |
| Page Template | Same as Section Template | — |
| Composition | draft→active, active→archived | archived→active (unless product allows) |
| Build Plan | pending_review→approved|rejected, approved→in_progress→completed|superseded | completed→pending_review, rejected→approved |
| AI Run | pending_generation→completed|failed_validation|failed | completed→pending_generation, any→pending_generation |
| Prompt Pack | draft→active, active→inactive|deprecated | deprecated→active (new version instead) |
| Documentation | draft→active, active→archived | archived→active (unless product allows) |
| Version Snapshot | active→superseded | superseded→active |

## 7. Lifecycle-transition checklist

- [ ] Section Template: draft→active, active→inactive/deprecated allowed; deprecated→active not allowed without policy.
- [ ] Page Template: same as Section Template.
- [ ] Composition: draft→active, active→archived; validation_status independent of status.
- [ ] Build Plan: pending_review→approved|rejected; approved→in_progress→completed|superseded; no reverse from completed/rejected to pending_review.
- [ ] AI Run: pending_generation→completed|failed_validation|failed only; no status rollback.
- [ ] Prompt Pack: draft→active, active→inactive|deprecated; new version for new active.
- [ ] Documentation: draft→active, active→archived.
- [ ] Version Snapshot: active→superseded when new snapshot for same scope created.
- [ ] Invalid: any transition not listed in §5 is disallowed (e.g. completed→draft, superseded→active).
