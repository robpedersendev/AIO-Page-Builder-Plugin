# Executor Locking and Idempotency Contract

**Spec**: §11.3 Job Queue Table(s); §39.8 Safe Failure Boundaries; §40.2 Single-Action Execution Flow; §40.3 Bulk Execution Flow; §40.5 Idempotency Rules; §40.6 Retry Rules; §40.7 Partial Failure Rules; §42.1 Queue Purpose; §42.2 Queueable Job Types; §42.6 Queue State Model; §42.7 Dead Job Recovery; §42.8 Concurrency Rules; §42.9 Locking Rules; §59.10 Execution Engine Phase

**Status**: Contract definition only. No executor or queue runner implementation. All execution and queue code must conform to this contract to avoid double-run, race conditions, and corrupted state.

**Upstream**: execution-action-contract.md (action envelope, result shape). Queue table: `aio_job_queue` (job_ref, job_type, queue_status, retry_count, lock_token, failure_reason, etc.).

---

## 1. Purpose

This contract formalizes **locking**, **idempotency**, **retry**, and **duplicate-suppression** rules for executor actions. Job/action lock ownership, idempotency keys, duplicate detection, retry eligibility, stale-lock handling, and partial-success reporting are defined so execution cannot corrupt state through double-run or race conditions. Lock and idempotency state are system-managed and never client-authoritative.

---

## 2. Principles

- **Execution must be deterministic where possible.** Same approved input and context must yield the same outcome or a defined refusal.
- **Duplicate actions must not silently run twice.** Same-target, same-action, same-context requests must be suppressed or deduplicated.
- **Partial success must be logged as partial success**, not full success (spec §40.7).
- **Queue and lock state must be inspectable** (spec §42.10). Stale locks must be detectable and recoverable (spec §42.7).
- **WordPress requests are not assumed serialized.** Locking is required where concurrent execution could affect the same target.

---

## 3. Lock Scope and Ownership

### 3.1 Lock scopes

Locks apply at one or more of these scopes. Each scope has a **scope key** used for lock acquisition and conflict detection.

| Scope            | Description                     | Scope key format example                    | Use case                          |
|------------------|----------------------------------|---------------------------------------------|-----------------------------------|
| **Job**          | Single queue row                | `job:{job_ref}`                             | Prevent same job running twice   |
| **Action**       | Single execution action         | `action:{action_id}`                        | Tie lock to action envelope      |
| **Plan**         | Build Plan                      | `plan:{plan_id}`                            | Serialize execution per plan     |
| **Plan item**    | Build Plan item                 | `plan_item:{plan_id}:{plan_item_id}`        | One execution per item at a time |
| **Target object**| Page, menu, token set           | `page:{post_id}`, `menu:{term_id}`, etc.    | Prevent concurrent mutation      |

**Combined scope**: A job may hold multiple scope keys (e.g. job + plan + target page). All must be acquired for the job to run; release applies to all.

### 3.2 Lock ownership

- **Owner**: The job record (or execution attempt) that holds the lock. Stored as `lock_token` in the queue table (spec §11.3). The token uniquely identifies the owner (e.g. `{job_ref}:{attempt_id}` or UUID).
- **Acquisition**: Before performing mutation, the executor must acquire the lock for the relevant scope(s). Acquisition fails if another owner holds the lock and it is not stale.
- **Release**: On completion (success, failure, or refusal), the executor must release the lock so the same scope can be used by another job or retry.
- **No client-supplied lock tokens.** Lock tokens are generated by the system when a job is claimed or an action is started.

### 3.3 Queue table fields (alignment)

The job queue table (`aio_job_queue`) includes:

| Field            | Purpose                                                |
|------------------|--------------------------------------------------------|
| `job_ref`        | Unique job identifier (idempotency key candidate).     |
| `queue_status`   | Current state (§4).                                    |
| `lock_token`     | Owner token when job is running; NULL when not held.   |
| `retry_count`    | Number of retry attempts.                              |
| `failure_reason` | Last failure reason (and/or error code).               |
| `started_at`     | When the job last started (for stale detection).      |
| `completed_at`   | When the job completed (success or terminal failure).   |

Lock state is implied: `lock_token` non-NULL and `queue_status` = running/retrying means the job holds the lock. Clear `lock_token` on release or when marking failed/dead.

## 4. Lock and Queue States

### 4.1 Queue status values (spec §42.6)

| Status      | Meaning                                                                 |
|-------------|-------------------------------------------------------------------------|
| `queued`    | Job is waiting to run. No lock held.                                   |
| `scheduled` | Job is scheduled for a future time. No lock held.                       |
| `running`   | Job is executing. Lock(s) held; `lock_token` set; `started_at` set.     |
| `retrying`  | Job failed and is scheduled for retry. Lock released between attempts.  |
| `completed` | Job finished successfully. Lock released.                               |
| `failed`    | Job finished with non-retryable failure or exceeded retries. Lock released. |
| `cancelled` | Job was cancelled. Lock released.                                      |
| `dead`      | Job is stalled or abandoned; stale lock recovered. Lock released.       |

**Stored in**: `queue_status` column. States are mutually exclusive.

### 4.2 Lock state (derived)

- **Unlocked**: `lock_token` is NULL (or empty). Scope is available for acquisition.
- **Locked**: `lock_token` is set; the owning job is in `running` (or, during handover, briefly in another state). Scope is not available for another job until released or declared stale.
- **Stale**: Lock is held but the owner has exceeded the maximum allowed execution time or the owner job is in a state that implies abandonment (§5.3).

## 5. Lock Acquisition and Stale-Lock Policy

### 5.1 Acquisition rules

1. **Before mutation**: Executor must acquire lock for the job (and, per action type, for plan and/or target object) before performing any site mutation.
2. **Single owner per scope**: Only one lock_token may hold a given scope key at a time (except where contract explicitly allows shared read locks; this contract defines exclusive locks only).
3. **Fail fast**: If lock cannot be acquired (scope held by another non-stale owner), the executor must not run. Return a result with status `refused` and error code `conflict` (or a dedicated `lock_not_acquired` if added to execution-action-contract).
4. **Atomicity**: Acquisition should be atomic where possible (e.g. UPDATE queue row SET lock_token, started_at, queue_status WHERE job_ref = ? AND (lock_token IS NULL OR stale) AND queue_status IN ('queued','retrying')).

### 5.2 Release rules

1. **On completion**: Clear `lock_token`, set `completed_at` or `failure_reason` as appropriate, set `queue_status` to `completed`, `failed`, or `cancelled`.
2. **On refusal**: Clear any lock that was acquired for this attempt; do not leave the job in `running` with lock held if no mutation will occur.
3. **On retry**: Release lock at end of each attempt. Next attempt re-acquires when the job is moved to `running` again.

### 5.3 Stale-lock detection and recovery

| Condition | Stale? | Recovery action |
|-----------|--------|-----------------|
| `queue_status` = `running` and `started_at` older than **max_run_seconds** | Yes | Mark job `dead` or `failed`; clear `lock_token`; set `failure_reason` to "stale lock" or equivalent. Scope becomes available. |
| Job runner process died (e.g. no heartbeat) | Policy-dependent | Admin or cron can mark job `dead` and clear lock after a timeout. |
| `queue_status` = `running` but job_ref not in current runner set | Yes (if verified) | Same as above. |

**Max run seconds**: Contract defines a constant or configurable value (e.g. 600 seconds for a single action). Jobs that run longer must be considered stale unless extended by a heartbeat.

**Recovery is capability-gated.** Only processes or users with permission to manage the queue may clear stale locks or move jobs to `dead`. Lock state is never taken from the client.

### 5.4 Stale-lock scenario matrix

| Scenario | Job A state | Job B same scope | Allowed? | Notes |
|----------|-------------|------------------|----------|--------|
| A running, B queued | running, lock_token set | queued | B waits | B cannot acquire until A releases. |
| A running (stale), B queued | running, started_at > max_run_seconds ago | queued | B may acquire after A marked dead | Recovery clears A's lock; B can then run. |
| A completed | completed, lock_token NULL | queued | B may acquire | Scope free. |
| A failed | failed, lock_token NULL | queued | B may acquire | Scope free. |
| A running, B different scope | running on page 1 | queued for page 2 | B may acquire | No scope overlap. |

## 6. Idempotency Key Strategy

### 6.1 Idempotency key requirements

- **Source**: The idempotency key is derived server-side from the **action envelope** and/or the **queue job**. It is never supplied authoritatively by the client (execution-action-contract §4.2 allows optional `idempotency_key` in the envelope; when present it may be used as input to derivation, but the system may override or normalize it).
- **Uniqueness**: For a given action type, plan_id, plan_item_id (if item-scoped), and target reference, the same logical action must produce the same idempotency key so duplicate submissions can be detected.
- **Storage**: The key can be stored in the job row (e.g. `job_ref` is the idempotency key for the job), or in a dedicated column if the table has one. The queue table uses `job_ref` as the unique job identifier; idempotency for "same action" can be implemented by (a) using a deterministic `job_ref` from plan_id + plan_item_id + action_type + target ref hash, or (b) checking for an existing job with the same idempotency key before enqueueing.

### 6.2 Recommended idempotency key format

- **Job-level**: `job_ref` = unique per job instance (e.g. UUID or `exec_{plan_id}_{plan_item_id}_{action_type}_{short_hash}_{timestamp}` for first submission).
- **Deduplication key** (for duplicate suppression): Deterministic from `plan_id` + `plan_item_id` + `action_type` + normalized target reference (e.g. post_id or internal_key). Example: `idem:{plan_id}:{plan_item_id}:{action_type}:{target_hash}`.

When a new execution is requested, the system may:
1. Compute the deduplication key from the envelope.
2. Look up an existing job with that key (or with same plan_id, plan_item_id, action_type, target) in a non-terminal state (`queued`, `scheduled`, `running`, `retrying`).
3. If found and not completed: **suppress** the duplicate (return existing job ref or "already queued" result). Do not create a second job.
4. If found and completed: Optional "already completed" handling — return prior result instead of re-running (§6.3).

### 6.3 Duplicate suppression behavior

| Case | Same plan_id | Same plan_item_id | Same action_type | Same target | Existing job state | Behavior |
|------|--------------|-------------------|------------------|------------|--------------------|----------|
| 1    | Yes          | Yes               | Yes              | Yes        | queued / running / retrying | **Suppress**: Do not enqueue a second job. Return "already in progress" or existing job_ref. |
| 2    | Yes          | Yes               | Yes              | Yes        | completed          | **Optional**: Return "already completed" and prior result; do not re-run (idempotent). Or allow re-run if policy permits "run again". |
| 3    | Yes          | Yes               | Yes              | Yes        | failed / dead      | **Allow new job**: Previous attempt terminated. New job is a retry or new attempt. |
| 4    | Different    | -                 | -                | -          | any                | No duplicate; different plan. |
| 5    | Yes          | Different         | -                | -          | any                | No duplicate; different item. |
| 6    | Yes          | Yes               | Yes              | Different  | any                | No duplicate; different target (e.g. different page). |

**Same target** is defined per action type: for `create_page` it might be plan_item_id (only one create per item). For `replace_page` it is the page_ref. For `update_menu` it is the menu_ref. Contract defines "same target" as equality of the normalized target reference used in the idempotency key.

## 7. Retry Eligibility and Retry Rules

### 7.1 Retry-eligible versus non-retryable failures

| Failure class | Retryable? | Examples |
|---------------|------------|----------|
| **Transient** | Yes       | Temporary DB error, timeout without mutation, rate limit, network blip. |
| **Dependency**| Deferred  | Dependency not satisfied yet (e.g. parent page not created). Retry later when dependency exists. |
| **Permanent** | No        | Invalid envelope, unauthorized, target not found, validation error, conflict (duplicate). |
| **Mutation already attempted** | No | Any failure after mutation has started (partial or full) — do not retry blindly; may require rollback or manual fix. |

Spec §40.6: Retry rules shall include retry eligibility, maximum retry count, distinction between automatic and manual retry, preservation of prior failure history, no silent infinite retry loop.

### 7.2 Retry eligibility scenario matrix

| Scenario | Error code / cause | Retry eligible? | Max auto-retries | Notes |
|----------|--------------------|-----------------|------------------|-------|
| DB deadlock or transient wpdb error | execution_failed (transient) | Yes | 3 | Increment retry_count; reschedule. |
| Dependency not satisfied | dependency_failed | Deferred | 0 (reschedule, not count as retry) | Re-queue for later; do not increment retry_count as failure. |
| Invalid envelope | invalid_envelope | No | 0 | Mark failed; do not retry. |
| Unauthorized | unauthorized | No | 0 | Mark failed. |
| Target not found | target_not_found | No | 0 | Mark failed. |
| Lock not acquired (conflict) | conflict | Deferred or retry | 1–2 | May retry once lock is free. |
| Snapshot required missing | snapshot_required | No | 0 | Fix then manual retry. |
| Stale lock recovered | - | N/A | - | Job marked dead; new job can be enqueued. |
| Partial success | execution_failed (partial) | No (automatic) | 0 | Manual intervention or separate rollback/retry. |

### 7.3 Maximum retry count and backoff

- **Maximum retry count**: Per job type or global (e.g. 3). Stored in `retry_count`; when `retry_count` >= max, next failure moves job to `failed`, not `retrying`.
- **Backoff**: Delay before retry (e.g. exponential or fixed). Contract does not mandate a specific backoff; implementation may use WP-Cron or queue scheduler.
- **Manual retry**: Admin or user with capability may create a **new** job (new job_ref) for the same action after fixing the cause, or explicitly "retry" which may create a new job or reset retry_count per policy. Prior failure history is preserved in logs.

## 8. Partial Success and Partial Failure

### 8.1 Partial-success semantics

- **Definition**: A multi-part action (e.g. bulk update menu with 10 items) where some parts succeed and some fail.
- **Result**: Status must be `partial` (execution-action-contract §10). Result object must include `error.partial_succeeded` and `error.partial_failed`.
- **Build Plan state**: Update only the items that succeeded to completed; leave failed items in a failed or retry state. Do not mark the whole step as completed (spec §40.7).
- **Logging**: Log partial success as partial; do not log as full success. Preserve logs and snapshots for failed parts.

### 8.2 Partial-failure scenario matrix

| Scenario | Action type | Outcome | Status | Build Plan update |
|----------|-------------|---------|--------|-------------------|
| Bulk menu: 8/10 items applied | update_menu | 8 succeeded, 2 failed | partial | Mark 8 items completed; 2 items failed. |
| Create page: page created but meta save failed | create_page | Page exists; meta incomplete | partial | Mark item in_progress or failed; record partial_succeeded (post_id), partial_failed (meta). |
| Token set: 5/5 applied | apply_token_set | All succeeded | completed | Mark all items completed. |
| Replace page: exception before any write | replace_page | No mutation | failed | No plan update for item; job failed. |

### 8.3 Tie-in to Build Plan state

- **On completed**: Update Build Plan item status to completed (or equivalent); update plan-level execution state if applicable.
- **On failed**: Update item status to failed; set failure_reason in plan or execution log.
- **On partial**: Update each succeeded item to completed; each failed item to failed; do not mark step as complete if any item failed.
- **On refused**: No Build Plan state change (no mutation occurred).

Lock release and queue status update happen regardless of outcome so that the same scope can be used for retry or another job.

## 9. Scenario Matrices Summary

### 9.1 Duplicate-action matrix

| Request | Existing job (same dedup key) | Behavior |
|---------|-------------------------------|----------|
| New execution request | None | Enqueue new job; acquire lock when run. |
| New execution request | Queued | Suppress; return "already queued" or job_ref. |
| New execution request | Running | Suppress; return "in progress". |
| New execution request | Retrying | Suppress; return "retry scheduled". |
| New execution request | Completed | Optional: return prior result (idempotent) or allow new job. |
| New execution request | Failed / Dead | Allow new job (new attempt). |

### 9.2 Stale-lock matrix

| Job A | Job B | Action |
|-------|-------|--------|
| Running, not stale | B wants same scope | B waits or gets "conflict". |
| Running, stale (started_at > max_run_seconds) | - | Recovery: set A to dead, clear lock. |
| Running, stale | B wants same scope | After recovery, B can acquire and run. |
| Completed / Failed | B wants same scope | B acquires (scope free). |

### 9.3 Retry eligibility matrix

| Failure | Retry? | Next state |
|---------|--------|------------|
| Transient (e.g. DB) | Yes (if retry_count < max) | retrying → reschedule. |
| Permanent (e.g. invalid_envelope) | No | failed. |
| Dependency not satisfied | Deferred | Re-queue; no retry_count increment. |
| Lock conflict | Optional retry | retrying or re-queue. |
| Partial success | No auto retry | completed with partial; manual if needed. |

## 10. Cross-References

- **Execution action contract**: Action envelope includes optional `idempotency_key`; result shape includes `status` and `error`. Refusal and failure both require release of any acquired lock.
- **Queue table**: `aio_job_queue` with `job_ref`, `job_type`, `queue_status`, `retry_count`, `lock_token`, `failure_reason`, `started_at`, `completed_at`. Implementation must align with this contract.
- **Build Plan**: Item and plan status updates after execution are defined in §8. Execution does not change plan state arbitrarily; only as outcome of a completed/failed/partial run.
- **Capabilities**: Lock reset, manual retry, and marking jobs dead are capability-gated. Do not expose raw lock tokens or internal keys to unauthorized users.
